## 0. 一句话

我们建的是**一台引擎**：把一个剧本数字化成"**时序、多视角的知识图谱**"，在它之上长出两类产品——**创作侧（上帝视角：提炼框架/查逻辑/补血肉/可视化）** 与 **游玩侧（角色视角：演绎/对抗/被审问，结构性不泄密）**。二者是同一份图的两个**视图**，不是岔路。

## 0.5 实现现状（2026-06 · 双模式 v4）

§1「一引擎两面」已**落地为界面的一级结构**——顶部「现场 ⇄ 上帝」切换 + 底部一根「幕」时间轴统驭一切：

- **🎭 现场 · 入局（玩家·迷雾）**：关系网当主舞台（点人 = 成为 TA、进雾、ego 聚焦）；右栏**认知卡**=TA 本幕的 线索（含 OCR 真实卡面内容）/ 信念真伪 / 守秘 / 目标台词 / 审问对话（防泄漏 agent）。→ 对抗 agent 的 demo 面。
- **🪄 上帝 · 创作（创作者·全知）= 创作画布**：把本子拆成 **68 事件 + 92 因果边**的故事图。L1 总览（幕×角色泳道+卡片+因果边）→ L2 钻取详图（一个事件=前因左/焦点中/后果右，ELK 分层）：**因果邻域**(读懂)/**⚡推演下游**(改它的涟漪逐波点亮+影响范围)/**🔎溯源上游**(追根因)，含**焦点锚定+导航栈+「你在这里」地图**防迷路。→ 改本/丰富剧情/查逻辑/看涟漪的 demo 面（求解器精确报警=下一步）。**认知矩阵降为可选镜头**。详见 `创作画布研究与设计.md`。
- **底部幕轴**：滑到哪幕全部视图更新，并显示每幕「发牌密度」（揭示事实+发放线索）。

**与本文早期设想的差异（重要）**：曾短暂把"认知矩阵"当成唯一 hero，经用户体验反馈否决——矩阵太抽象、不适合做玩家第一眼。结论：**矩阵是创作者的分析工具（归上帝模式），关系网才是玩家的直观入口（现场模式）**。研究依据与 roadmap 见 `可视化研究与方案.md`（其 §1「矩阵当主视图」结论已据此修订，§2 数据模型地基仍是后续主线）。

运行/Demo 走法见 `README.md`。代码已实现：`shared/`(Zod+sliceGraph) · `backend/`(compile/chat/dossier/clues/server, Hono, **DeepSeek**) · `frontend/`(Vite+React19+React Flow+自绘 SVG 矩阵+Tailwind+Zustand)。

## 1. 核心架构判断：一台引擎，两个视图

| | **上帝视角（创作侧）= 角度1** | **角色视角（游玩侧）= 角度2** |
|---|---|---|
| 看到什么 | 完整真相图 + 结构/可解性 | 自己那块带迷雾的子图 |
| 用途 | 提炼 SOP 框架、查逻辑、补血肉、可视化 | 演绎 / 对抗 / 被审问，守秘密 |
| 数据来源 | `02_truth` + `timeline` + 全量 facts/clues | `03_characters/<角色>/knowledge·goals·relations` |

→ 不必在 A/B 里二选一来定**数据底座**；只需决定先 demo 哪个**视图**。

---

## 2. 核心抽象：`G(幕, 视角)`

整个引擎围绕一个对象：

> **`G(act, perspective)` → 一张子图。**

- **Z 维（幕/时间）**：每个 事实/线索/关系 都带"第N幕起可知"属性，图**沿 Z 生长**。
  - 数据已就位：`00_index/timeline.md`（事件×年/幕×涉及人物×事实ID）、`goals_by_act`、`perceives_by_act`、各幕 `[reveals: F0xx]`。
- **XY 维（某幕切片）**：那一刻谁知道什么、关系如何、有哪些线索在桌上。
- **视角维**：
  - `G(act, 上帝)` = 截至该幕的全部真相（裁判持有）。
  - `G(act, 角色)` = 该角色截至该幕"知道/相信"的子图（含假信念）。
  - **隔离墙 = `G(act,上帝) \ G(act,角色)`**（例：「她杀的延迟=阿奇」在任何 `G(act,黛利拉)` 中都不存在）。

**这一个抽象同时定义了三件事：**
- 进度设计 → Z 由"幕引擎"推进，每推进一幕解锁图的下一层。
- 数据流 → 每次提问/agent 回合都参数化为 `(当前幕, 视角)`，agent 上下文 = `G(幕,角色)`。
- UI → 渲染 `G(幕,视角)` + 一个**幕滑块(Z)** + 一个**视角切换器**。

---

## 3. 图数据模型（引擎内部）

节点/边沿用 `00_index/本体schema初稿.md`，关键是给每条加 **时序/视角** 属性：

**节点**：Character / Fact / Clue / Puzzle / Act / NPCPerformance / Event / Org / Prop
**边（每条可带 `from_act` / `perspective` / `reveals` 属性）**：
- 知识边（反泄漏核心）：`KNOWS` · `BELIEVES_FALSELY` · `MUST_HIDE{from,until}`
- 剧情边：`REVEALS`(Clue→Fact) · `TRIGGERS`(Act→Clue|NPCPerformance) · `INVOLVES`(Event→Character) · `CP|KIN|ENEMY|ALIAS_OF|MEMBER_OF`
- 真相边：`SURFACE_OF` · `CONTRADICTS` · `CROSS_REF`

**编译产物**：一个构建步骤把 `digitized/` 的 md/yaml/csv 解析成**单一 graph JSON**（节点+边+act/perspective 元数据），驱动前端与运行时。POC 已验证可解析（`_poc/`，20 实体/200 事实/246 信念/线索可达性）。

## 4. Demo 主体：时序叙事图谱浏览器

一个界面同时展示 资产 + 时间模型 + 反泄漏 + 创作审稿 + 活的 agent。

```
┌─────────────────────────────────────────────────────────────┐
│ 《流氓叙事》叙事图谱     视角:[上帝▼/黛利拉/程聿怀…]  [▶对话]  │
│ 幕: 序─①─②─[③]─④─⑤─⑥─⑦   ◀━━━━●━━━▶  (Z 滑块)            │
├──────────────────────────────────┬──────────────────────────┤
│   ● 羌青瓷━cp(真)━● 程聿怀        │  ▸ 选中: 黛利拉           │
│       ┃秘密            ┃          │  本幕知道: 24 真信念      │
│   ● 伯纳德    阿奇●╌╌假信念╌● 黛  │  假信念: 阿奇还活着 ★    │
│   (灰=她视角看不到的真相节点)     │  秘密: 帮"延迟"杀台长     │
│   线索→事实 虚线; 已解锁=亮       │  [与黛利拉对话(本幕)]→    │
└──────────────────────────────────┴──────────────────────────┘
```

**核心交互**
1. **幕滑块(Z)**：拖动看图随剧情生长，事实/线索逐幕点亮。
2. **视角切换**：切到某角色，整图大半**进雾变灰**——隔离墙一眼可见（卖反泄漏）。
3. **节点检视**：点角色看其 KB/goals/relations@本幕；点事实看哪些线索揭示它、何时；点线索看 reveal 目标。
4. **与 agent 对话**：从角色节点@第N幕开对话，agent 用 `G(幕,角色)` 编译，**旁边摆其子图="它只知道这些"**，套不出墙后真相。
5. **创作侧叠加层（上帝视角）**：框架骨架 + **真相解锁/张力曲线**（每幕可知真相%）+ 自动审稿旗标。

## 5. 运行时 / 数据流（对抗状态机）

把对抗设计成 Z 上的状态机：

- **GameState** = { 当前幕, 世界状态, 每角色 KB 切片=`G(幕,角色)`, 已揭示事实, 关系状态, 可用线索 }。
- **导演/裁判引擎**：持有 GameState 与幕流程（直接用 `05_acts/*` 的触发/发线索/演绎时点）。
- **一个回合**：
  1. 玩家/主持抛 提问/动作 → 路由到相关 agent。
  2. 每个 agent 拿 `G(幕,角色) + reveal策略 + goals_by_act + 输入` → 产出回应。
  3. **裁判校验**：没泄露 `∉G(幕,角色)` 的事实；不与"主线锁定层=`02_truth`"矛盾。
  4. 通过则提交、更新状态；命中触发则**推进 Z**（解锁下一层、注入新线索/演绎）。
- **主线锁定层** = 固定真相图（`02_truth`+`timeline`）；生成的新分支须对它 **diff 校验通过**才出栈 → demo 里展示绿色"主线未被触碰 ✓"。
- 对应原则：**固定骨架（02_truth + 幕骨架）、即兴血肉（agent 在 `G(幕,角色)` 内的对话）**。

## 6. 技术栈与仓库结构（已收敛）

**总原则**：前后端**显式分离**为两个独立项目，靠 HTTP 通信、靠 `shared/` 共享图类型；App 全 **TypeScript**；Python 数字化流水线**本阶段不写**（本子已数字化完；graph 编译改用 TS 写在 backend，读现成 `digitized/`，全程零 Python）。

仓库结构（pnpm workspaces monorepo）：

    repo/
    ├── frontend/   Vite + React 19 + React Flow + 自绘SVG矩阵 + Tailwind + Zustand   ← 纯 UI
    ├── backend/    Node + Hono（compile/chat/dossier/clues/server）                  ← graph 编译/查询 + agent对话
    ├── shared/     Zod schema + sliceGraph（图模型）                                  ← 前后端单一事实源
    └── digitized/  数据（已有）+ _ocr（PDF OCR 文本，喂线索卡面内容）

- **前端 = Vite + React 19**（不用 Next.js）：现场=**关系网 React Flow**；上帝=**总览画布自绘 SVG**（幕×角色泳道因果图）+ **钻取详图 React Flow + elkjs 分层**（因果路径任务 node-link 最优，见 `创作画布研究与设计.md` §4）；认知矩阵自绘 SVG 留作可选镜头；Tailwind 暗色风；Zustand 做跨视图选择总线（brushing/linking）。
- **后端 = Node + Hono**：轻、TS 原生、随处部署；暴露 `GET /graph`、`GET /character/:name`、`GET /clue/:id`、`GET /act/:ord`、`POST /chat` 等；将来有状态多 agent session / websocket 直接往里加，不搬家。
- **LLM = DeepSeek**（OpenAI 兼容端点，直接 fetch，未用 AI SDK）：`deepseek-chat`(flash) 默认、可切 `deepseek-reasoner`(pro)；key 在 `.env`（不入库）。当初设想的 Anthropic/AI SDK 未采用——直连更省依赖。
- **共享 = Zod**：图 节点/边/`G(幕,视角)`/AgentState 的 schema 与类型，前后端共用、加载时校验。
- **分层托管**：前端→**Vercel**；后端→**本地起步，长大后 Render / Azure**（账号已有）。
- **数据库 = 零数据库**：`digitized/`→内存图，filter 出 `G(幕,视角)`。何时上图库(Neo4j/Kùzu)：多本子库 / 持久化多人 session / 服务端重图算法。向量库(pgvector)：仅当 agent 需对长正文做 RAG 时。

> 决策依据：见根目录对话——"分离是代码结构决策、托管是另一回事"；脚手架由工具生成，分离的设置成本不落在用户身上，故选清晰分离 + 可成长，而非 Next.js 全栈图快。

## 7. 高价值特性清单（含新增点）

1. **Fog-of-War 做核心视觉语言**：每角色 `G(幕,角色)` 渲成半雾图——反泄漏从"日志一行"变"一眼可见"，最强差异化。
2. **张力/解锁曲线**（创作侧）：每幕"可知真相%"成曲线，作者看揭示节奏。数据独有。
3. **What-if 沙盒 + 主线 diff 护栏**：抛动作→agent 反应→亮"主线完整✓"，安全展示对抗生成力。
4. **NPC 演绎 agent = 风险最低的活 demo 切口**：全脚本（verbatim 台词+触发，见 `04_npcs/*/演绎脚本.md`、`05_acts`），让 AI 接管羌青瓷火柴疗法、扛玩家即兴；直接对接"替演员降成本"。
5. **一致性/可解性审稿器**：已自动抓出 伯纳德/卢特、1995vs2000（见 `_QA/`）→ 包成创作"逻辑检查"。
6. **跨本子 SOP 框架**：创作工具真正威力，**需第 2 个本子**才显现 → 早规划。

## 9. 建议 v1 范围（草案，待第8节决策后定稿）

**v1 = 时序叙事图谱浏览器（主体）+ 单角色对话活切片**，分两个里程碑：

- **里程碑①（零 LLM / 零 DB）· 图谱浏览器** ← 无任何待决策即可开工

  - frontend：React Flow 渲染；幕滑块(Z) + 视角切换(上帝⇄角色) + 节点检视 + fog-of-war 雾化。
  - 创作叠加：张力/解锁曲线 + 审稿旗标（复用 `_QA` 结论）。
  - **本地即可演示；前端可静态部署 Vercel。**
  - 注：浏览器同时承载"上帝视角(创作侧)"与"角色迷雾(游玩侧)"——是两个产品角度的**共享底座**，故先做它不必先选 §8.1 的"哪半边"。

**后续**：多 agent 对抗裁判循环（机机/人机，需 §8.2）→ What-if 沙盒 → NPC 演绎 agent → 第二个本子上 SOP 框架。
